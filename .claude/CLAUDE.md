# Claude Code 設定

## 必須ルール

- @../CONTRIBUTING.md の内容に必ず従うこと
- cd でのディレクトリ移動は `&&` で繋げずに別コマンドで行うこと
- コミットメッセージは英語で記述する
- コミット前に `cargo fmt` すること
- コミットメッセージのフッターに Claude の署名（Co-Authored-By）を付けないこと
- マルチステップの実装タスクでは、各ステップ完了後にコミットすること（全変更を最後にまとめてコミットしない）
- コミット時に `$()` コマンド置換や `echo ... | git commit -F -` を使わないこと（承認ダイアログが表示されるため）。代わりに `-m` オプションで直接渡す：
  ```bash
  git commit -m 'feat: add feature

  detailed description'
  ```
- 複数のコマンドを `&&` で連結しないこと。代わりに個別の Bash ツール呼び出しを使う

# コードレビューワークフロー

ユーザからコードレビュー依頼を受けた場合、Agent Teams を使ってレビュアーチームを構成する。

## 手順

1. `git diff` で変更スコープを分析
2. スケーリングルールに基づきレビュアーを選択
3. `TeamCreate` でレビューチームを作成（チーム名: `code-review`）
4. 選択したレビュアーごとに `TaskCreate` でレビュータスクを作成
5. 選択したレビュアーを **1つのメッセージで並列起動**（Agent tool, `team_name: "code-review"`, subagent_type 指定、**name パラメータ必須**、プロンプトにピアディスカッション指示を含める — 「ピアツーピアディスカッション」セクション参照）
6. 各レビュアーの初期結果を収集（この時点でエージェント間のピアメッセージは交換済み）
7. ピアディスカッションラウンドを実施（条件を満たす場合。「ピアツーピアディスカッション」セクション参照）
8. 各レビュアーの最終結果を統合フォーマットで出力
9. レビュアーに `shutdown_request` を SendMessage で送信し、`TeamDelete` でクリーンアップ

## スケーリングルール

| 条件 | 起動するレビュアー |
|---|---|
| 3ファイル以下 | 変更内容に関連するレビュアーのみ（最低2つ） |
| 4ファイル以上 | 4レビュアー全て |
| 数学/金融ロジック変更 | financial-correctness-reviewer を必ず含める |
| blockchain/wallet 変更 | security-reviewer を必ず含める |

## レビュアー選択ガイド

- **financial-correctness-reviewer**: 算術演算、BigDecimal、NearValue、価格/レート計算、AMM数式、手数料ロジック、価格の予測
- **security-reviewer**: blockchain RPC、ウォレット操作、鍵管理、トランザクション実行、外部入力パース、SQLクエリ、ログの機密情報
- **rust-quality-reviewer**: Rust コード変更全般（設定/ドキュメントのみの変更は除外可）
- **architecture-reviewer**: 複数クレートにまたがる変更、pub API変更、新モジュール追加、依存構造変更、テスト追加/変更

## ディスカッションラウンド

「ピアツーピアディスカッション」セクションの共通プロトコルに従う。

## クリーンアップ

統合レポート出力後、以下を実行:
1. 全レビュアーに `shutdown_request` を SendMessage で送信
2. 全レビュアーの終了を確認後、`TeamDelete` でチームリソースを削除

## 統合出力フォーマット

```
# コードレビュー結果

## CRITICAL (N件)
### [ファイルパス]
- 🏦 **[金融正確性/批判的]** 指摘内容
- 🔒 **[セキュリティ/慎重]** 指摘内容

## WARNING (N件)
...

## SUGGESTION (N件)
...

## 良い設計判断 👍
...

## 総評
各レビュアーの視点を踏まえた総合評価。対立する意見がある場合は両論併記。
```

### 統合ルール
- 重複排除: 複数レビュアーが同じ問題を指摘 → 1エントリに統合し全視点を注記
- 重大度順ソート: CRITICAL → WARNING → SUGGESTION
- ファイル別グループ化: 各重大度レベル内でファイルパスごとに整理
- 対立意見の保存: レビュアー間で意見が分かれる場合は両論併記
- 肯定的評価の保持: architecture-reviewer の「良い設計判断」セクションを残す

# コード調査ワークフロー

ユーザからバグ調査、エラー追跡、原因特定、呼び出し元分析などの依頼を受けた場合、親 Claude が直接オーケストレーションする。

**判別基準**: diff が入力 → レビューワークフロー、質問/症状/エラーが入力 → 調査ワークフロー、実装依頼が入力 → 実装計画ワークフロー（複雑さ判定後）

## フェーズ1: スコープ特定

1. code-investigator を単独起動（Agent tool, subagent_type 指定）し、フェーズ1レポート形式でスコープを特定させる
2. 指示テンプレート: 「以下の問題を調査してください。調査チームのフェーズ1として、フェーズ1レポート形式で報告してください。問題: {ユーザの質問/症状}」
3. code-investigator の結果を確認し、フェーズ2が必要か判断する

**注意**: フェーズ1は単独起動のため Agent Teams は不要。subagent で十分。

## フェーズ2スキップ条件

以下の場合はフェーズ1だけで完結し、code-investigator の結果をそのまま報告する:
- 呼び出し元一覧、定義箇所の特定など単純な調査
- code-investigator が明確な原因を特定済みで、専門観点の深掘りが不要な場合
- フェーズ2推奨が空または「不要」の場合

## フェーズ2: 専門エージェントによる深掘り

フェーズ2ではディスカッションラウンドが必要になる可能性があるため、Agent Teams を使用する。

### 手順

1. `TeamCreate` で調査チームを作成（チーム名: `investigation`）
2. スケーリングルールに基づきエージェントを選択
3. 選択したエージェントごとに `TaskCreate` で調査タスクを作成
4. 選択したエージェントを **1つのメッセージで並列起動**（Agent tool, `team_name: "investigation"`, subagent_type 指定、**name パラメータ必須**、プロンプトにピアディスカッション指示を含める）
5. 各エージェントの初期結果を収集（この時点でエージェント間のピアメッセージは交換済み）
6. ピアディスカッションラウンドを実施（条件を満たす場合。「ピアツーピアディスカッション」セクション参照）
7. 統合レポートを出力
8. 全エージェントに `shutdown_request` を SendMessage で送信 → `TeamDelete`

### スケーリングルール

| フェーズ1で特定された領域 | 起動するエージェント |
|---|---|
| 算術/金融計算/AMM/手数料 | financial-correctness-reviewer |
| blockchain/RPC/wallet/トランザクション | security-reviewer |
| Rust コード品質問題 | rust-quality-reviewer |
| クレート間依存/設計問題 | architecture-reviewer |
| 判断に迷う場合 | code-investigator の推奨に従う（最低1つ） |

### エージェントへの指示テンプレート

各専門エージェントに以下を含めて指示する:
- フェーズ1の調査結果（関連ファイル、初期仮説）
- ユーザの元の質問/症状
- 専門観点での検証指示: 「あなたの専門観点（{セキュリティ/金融正確性/Rust品質/設計}）から、以下の仮説と関連コードを検証してください」

### ディスカッションラウンド

「ピアツーピアディスカッション」セクションの共通プロトコルに従う。

## 統合出力フォーマット


### 統合ルール

- 対立意見の保存: エージェント間で見解が分かれる場合は両論併記

# 実装計画ワークフロー

ユーザから実装依頼を受けた場合、変更の複雑さに基づいて Agent Teams による計画を実施するか判断する。

## 自動トリガー基準（複雑さベース）

**Agent Teams で計画する場合**（以下のいずれかを満たす）:
- 設計上のトレードオフや判断が必要（複数のアプローチが考えられる）
- 既存アーキテクチャへの影響が大きい（依存関係の変更、pub API の変更等）
- セキュリティや金融計算に関わる変更
- 新しいクレートやモジュールの追加

**通常の計画で十分な場合**（以下のいずれかを満たす）:
- 実装パスが明確で代替案がない
- 既存パターンの単純な拡張
- ユーザが詳細な計画を既に提示している
- 機械的な変更（リネーム、フォーマット等）

## フェーズ1: スコープ分析（単独エージェント、Teams不要）

1. `implementation-planner` を単独起動（Agent tool, subagent_type 指定）し、フェーズ1レポート形式でスコープを分析させる
2. 指示テンプレート: 「以下の実装を計画してください。計画チームのフェーズ1として、フェーズ1レポート形式で報告してください。要件: {ユーザの要件}」
3. フェーズ1レポートを確認し、フェーズ2の要否を判断する

**注意**: フェーズ1は単独起動のため Agent Teams は不要。subagent で十分。

## フェーズ2スキップ条件

以下の場合はフェーズ1だけで完結し、implementation-planner の結果をそのまま計画として採用する:
- 実装パスが単純で明確
- リスク領域が特定されていない
- フェーズ2推奨が空または「不要」

## フェーズ2: 専門エージェントによる計画レビュー（Agent Teams）

フェーズ2ではディスカッションラウンドが必要になる可能性があるため、Agent Teams を使用する。

### 手順

1. `TeamCreate` で計画チームを作成（チーム名: `implementation-planning`）
2. スケーリングルールに基づきエージェントを選択
3. 選択したエージェントごとに `TaskCreate` で計画レビュータスクを作成
4. 選択したエージェントを **1つのメッセージで並列起動**（Agent tool, `team_name: "implementation-planning"`, subagent_type 指定、**name パラメータ必須**、プロンプトにピアディスカッション指示を含める）
5. 各エージェントの初期結果を収集（この時点でエージェント間のピアメッセージは交換済み）
6. ピアディスカッションラウンドを実施（条件を満たす場合。「ピアツーピアディスカッション」セクション参照）
7. 統合計画を出力
8. 全エージェントに `shutdown_request` を SendMessage で送信 → `TeamDelete`

### スケーリングルール

| 条件 | 起動するエージェント |
|---|---|
| デフォルト（2+クレート） | architecture-reviewer + 関連スペシャリスト（最低2つ） |
| 新クレート/モジュール追加 | architecture-reviewer 必須 |
| 金融/AMM ロジックの変更 | financial-correctness-reviewer 必須 |
| blockchain/wallet の変更 | security-reviewer 必須 |
| 重要な Rust API 設計 | rust-quality-reviewer |
| 4+クレートに影響 | 4スペシャリスト全員 |

### 専門エージェントへの指示テンプレート

```

1. 計画にリスクや見落としがないか検証
2. 実装上の制約やガードレールを提示
3. より良いアプローチがあれば提案

## ユーザの要件
{ユーザの元の要件}

## フェーズ1スコープ分析結果
{implementation-planner のレポート}
```

### ディスカッションラウンド

「ピアツーピアディスカッション」セクションの共通プロトコルに従う。

## クリーンアップ

統合計画出力後、以下を実行:
1. 全エージェントに `shutdown_request` を SendMessage で送信
2. 全エージェントの終了を確認後、`TeamDelete` でチームリソースを削除

## 統合出力フォーマット

```

## 概要

### 🔒 セキュリティ（該当する場合）
### 🏦 金融正確性（該当する場合）
### ⚡ Rust品質（該当する場合）

## 実装ステップ
### ステップ1: {概要}（コミット単位）
- ファイル / 内容 / 参考パターン / テスト

### ステップ2: ...

## 注意事項

```

### 統合ルール
- コードレビューと同様（重複排除、重大度順ソート、対立意見保存）
- フェーズ1のアプローチ推奨と各スペシャリストの評価を対応付け
- 各ステップ = 1コミット（CONTRIBUTING.md 準拠）

# 実装チェックポイント

実装計画ワークフローで生成された計画に基づいて実装する際、以下の2つのトリガーで専門エージェントを再活用できる。いずれも任意であり、Agent Teams は使わず単独 subagent で起動する。

**適用範囲**: 実装計画ワークフローで計画が生成された場合のみ。通常の実装には適用しない。

## トリガー1: ステップ内スポットチェック（コミット前）

### 条件
計画のステップにリスク注釈（`リスク: ... （推奨: {specialist}）`）が付いており、そのステップのコード変更が完了したがまだコミットしていない。

### スキップ条件
- ステップの変更が軽微（use 追加のみ、設定値変更のみ等）
- リスク領域に実際には影響しなかった

### 手順
1. 推奨された専門エージェントを単独 subagent として起動
2. `git diff` の出力とステップの概要を提供
3. CRITICAL 指摘があればコミット前に修正、WARNING/SUGGESTION は記録のみ

### 指示テンプレート

```
以下のステップの変更を、あなたの専門観点（{専門領域}）から簡潔に検証してください。

## 計画上のリスク
{計画のリスク注釈}

## ステップの概要
{計画のステップ説明}

## 変更差分
{git diff output}

CRITICAL の指摘のみ詳細に報告してください。WARNING/SUGGESTION は1行で簡潔に。
```

## トリガー2: 再計画（計画からの乖離時）

### 条件
以下のいずれかを満たす:
- ステップの実装が2回修正してもコンパイル/テストに通らない
- 計画の前提が誤っていたことが判明（API不在、依存サイクル、アプローチの根本的問題等）

### スキップ条件
- 修正が明白（typo、import 漏れ、単純な型不一致等）

### 手順
1. `implementation-planner` を単独 subagent として再起動（進捗チェック形式）
2. planner の報告を確認:
   - **修正版ステップが提示された場合**: 修正版計画に基づいて実装を継続
   - **「フル計画ワークフローの再実行を推奨」と報告された場合**: Plan モードに入り、実装計画ワークフローをフェーズ1から再実行する

### 指示テンプレート

```
実装中に問題が発生しました。進捗チェック形式で残りのステップを再計画してください。
修正が残りステップの微調整で済まない場合は、「フル計画ワークフローの再実行を推奨」と明記してください。

## 元の計画
{original plan}

## 完了済みステップ
{list of completed steps with commit hashes}

## 発生した問題
{description of the problem}

## 試したこと
{what was attempted to fix it}
```

# ピアツーピアディスカッション（共通プロトコル）

全ワークフロー（コードレビュー、コード調査、実装計画）のディスカッションラウンドに適用する共通プロトコル。
エージェント同士が直接 SendMessage でやりとりし、親の仲介を最小限にする。

## エージェント起動プロンプトへの追加指示

各エージェントのプロンプト末尾に以下を追加する:

```
## ピアディスカッション指示

チーム内の他のエージェント:
{ピアエージェント名と専門領域のリスト（例: "security-reviewer（セキュリティ）, architecture-reviewer（設計）"）}

### フェーズ1: 結果共有
分析完了後、以下を実行してください:
1. 自分の分析結果を要約（CRITICAL/WARNING/SUGGESTION 別、各1-2行）
2. SendMessage で各ピアエージェントに要約を送信
3. 送信完了後、親に初期結果を報告して待機

### フェーズ2: ピアレビュー（親からトリガーされた場合）
ピアから受信した結果を踏まえて:
1. ピアの指摘に対する補足や異なる見解があれば述べる
2. ピアの指摘を踏まえた新たな発見があれば追加する
3. 自分の専門外の疑問があれば、該当するピアに直接 SendMessage で質問する
4. 最終レポートとして報告する

### 選択的質問の指針
- セキュリティ面の懸念 → security-reviewer に質問
- 数値計算・金融ロジックの懸念 → financial-correctness-reviewer に質問
- Rust イディオム・品質の懸念 → rust-quality-reviewer に質問
- 設計・依存関係の懸念 → architecture-reviewer に質問
- 質問は具体的に（ファイルパス・行番号・コードスニペットを含める）
- 質問がない場合は送信不要
```

## ディスカッションフロー（親の手順）

1. 全エージェントの初期結果を収集する（この時点でエージェント間のピアメッセージは交換済み）
2. 実施条件を満たすか判定する
3. 条件を満たす場合、各エージェントに以下のトリガーを SendMessage で送信（並列）:
   ```
   ピアから受信した結果を確認してください。フェーズ2のピアレビューを実施し、最終レポートを提出してください。
   ```
4. 各エージェントの最終レポートを収集する（選択的質問のやりとりが完了するのを待つ）

## 実施条件

初期分析で2つ以上のエージェントが指摘を出した場合に実施する。

## スキップ条件

以下のいずれかを満たす場合、ディスカッションをスキップし初期結果をそのまま使用する:
- 起動エージェントが1つだけ
- 全エージェントが「指摘なし」
- 指摘ありのエージェントが1つだけ

# DB 接続について

- 基本的に DB への接続といえば run_local で動いているものを指す
- run_local の DB への接続は特に指定がない限り readonly ユーザを使う
